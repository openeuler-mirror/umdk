# verbs-to-urma-converter skill 设计与使用文档

> 对应 skill 版本: 1.0 | URMA API 版本: 25.12.0

---

## 1. 概述

### 1.1 定位

本 skill 是一个**知识驱动的代码迁移工具**，用于将 RDMA Verbs（libibverbs）代码自动转换为 URMA（统一远程内存访问）API 代码。它不包含可执行逻辑，而是一套结构化的参考文档体系，由 AI Agent 在迁移过程中按需查阅、逐步执行。

**能力范围**：

- 单文件或多文件 C 项目的 Verbs → URMA API 替换
- 数据结构、枚举、字段名的映射转换
- 连接模型的重构（QP modify_qp → import_jetty + bind_jetty）
- 地址交换协议的改写（LID+GID+QPN → EID+JPN）
- 编译验证与 patterns/pitfalls 全量审查

**不覆盖**：

- 非 C 语言项目
- Verbs 生态工具（如 `ibv_devinfo`、`rdma_*` 命令行工具）的迁移
- 运行时功能正确性测试（skill 止于编译通过 + 代码审查，不执行程序验证语义）

### 1.2 与手动迁移的区别

| 维度 | 手动迁移 | skill 辅助迁移 |
|------|---------|---------------|
| API 查找 | 逐个搜索头文件/文档 | 查阅 mapping.md 查找表 + 删除清单 |
| 代码编写 | 凭经验编写，易遗漏字段 | patterns.md 提供可复制的完整代码（含错误处理模式） |
| 错误预防 | 遇到问题再查 | pitfalls.md 提前规避 28 类已知陷阱 |
| 连接判断 | 凭经验判断 RC/RM/UM 流程 | mapping.md 决策树直接确定正确操作 |
| 验证 | 人工检查 | 交付物约束 + 编译验证 + 全量审查 |
| 一致性 | 因人而异 | 统一约束和输出格式 + 阶段交付物验收 |
| 效率 | 大型项目数天~数周 | 大幅缩短，但受 Agent 执行质量影响 |

## 2. 设计理念

### 2.1 为什么采用"知识库"而非"规则引擎/脚本"

Verbs → URMA 的迁移不是简单的 1:1 文本替换：

- **语义重构**：`ibv_modify_qp(RTR/RTS)` 不是替换一个函数，而是变成 `import_jetty` + `bind_jetty` 两个步骤，且 bind 仅 RC 模式需要
- **结构体重组**：`ibv_sge.lkey`（整数）→ `urma_sge.tseg`（指针），类型体系变了
- **概念消除**：PD（保护域）在 URMA 中隐式存在，需要删除而非替换
- **协议变更**：地址交换从 `lid:qpn:psn:gid` 变为 `jpn:eid`

这些转换需要理解上下文语义，脚本无法胜任，而 LLM Agent 可以阅读代码上下文后做出判断。知识库为 Agent 提供可靠的参考，避免其凭"记忆"编造不存在的 API。

### 2.2 定义"是什么"而非"怎么做"

skill v2.1 的核心设计原则：**约束结果（outcome），释放过程（process）**。

SKILL.md 作为编排层，应定义迁移的**交付物、约束、验收标准**——即"迁移完成时必须满足什么"，而非规定 Agent "用什么步骤、什么命令、什么格式达到"。原因：

1. **过程灵活性**：Agent 拥有自己的工具（Glob、Grep、Read、Edit、Bash 等），硬性规定 `find / mkdir / cp` 等具体命令既冗余又限制 Agent 发挥其能力
2. **结果稳定性**：只要交付物满足约束（如"编译通过""无 Verbs 残留""每个 pattern/pitfall 已检查"），Agent 用什么路径达到并不影响迁移质量
3. **减少指令冗余**：过于详细的步骤指令会导致 Agent 产生"遵守形式而非实质"的倾向——按步骤走过场，而不深入思考每步的目的
4. **适应不同项目**：项目规模和结构差异很大，固定步骤序列无法适应所有场景

**具体体现**：

| 维度 | v2.0（"怎么做"） | v2.1（"是什么"） |
|------|-----------------|-----------------|
| 阶段定义 | 编号步骤序列（1.1→1.2→1.3→1.4） | 目标 + 必须交付物 + 硬约束 |
| 文件操作 | 具体 bash 命令（`find`、`mkdir`、`cp`） | 信息需求描述 + 目录结构约束 |
| 验证清单 | 固定格式模板（`P-XX: PASS/FAIL/N/A`） | 验证完成条件（每项已检查、无未修复 FAIL） |
| 转换顺序 | 硬性序列（头文件→主源文件→工具文件→构建文件） | 灵活约束（头文件类型一致性、构建文件最后更新） |
| 检查点 | 完整 markdown 模板 | 必须包含的信息项 |
| 硬约束 | 散落在步骤描述中 | 集中呈现于工作流之前 |

### 2.3 为什么拆成 4 个参考文档

单一的大文件会导致 Agent 检索效率低下、容易遗漏章节。拆分后每个文档职责单一：

| 文档 | 职责 | Agent 使用场景 |
|------|------|---------------|
| `mapping.md` | **查找** — "X 的 URMA 等价物是什么？"；**判断** — "这个 Verbs 调用应该替换还是删除？"；**决策** — "RC/RM/UM 该用哪个流程？" | 遇到未知 API 时快速查找；遇到 modify_qp 时查决策树 |
| `patterns.md` | **实现** — "怎么写 URMA 版本的 X？" | 需要可复制的完整代码模式 |
| `pitfalls.md` | **防错** — "为什么 X 失败？" | 转换后检查是否踩坑 |
| `urma_sample.md` | **规范** — "完整的 URMA 应用长什么样？" | 仅当 patterns.md 不够详细时按需阅读（~1280 行） |

这四个文档的**信息粒度递减**：mapping 最细（逐条映射），patterns 中等（代码片段），pitfalls 较粗（分类陷阱），sample 最粗（完整示例）。Agent 可以按需深入，不必每次都读完所有内容。urma_sample.md 默认不加载，仅在 patterns.md 对特定 API 描述不够详细时才读取，以减少 context 消耗。

**参考文件本身就是声明式知识**——它们定义"映射关系是什么""正确做法是什么""要避免什么"，不规定 Agent 按什么顺序查阅、查多少次。这是好的设计，v2.1 不改变参考文件。

### 2.4 为什么 4 阶段都是强制性的

**阶段 1（准备）**：不分析源码就无法建立完整映射，遗漏 API 导致后续转换不完整。

**阶段 2（转换）**：核心产出阶段，必须有结构化的输出（验证结果）而非无序的代码。

**阶段 3（编译验证）**：编译通过是正确性的必要条件——未定义引用、类型不匹配等问题只能在编译阶段暴露。

**阶段 4（审查优化）**：这是**最关键也最容易被跳过**的阶段。许多问题编译无法捕获：

- 遗漏 `urma_ack_jfc()` → 资源泄漏，编译正常，运行时系统不稳定
- 遗漏 `urma_unimport_jetty()` → 资源泄漏，编译正常
- 访问标志组合错误（`LOCAL_ONLY | READ`）→ 编译正常，运行时失败
- `urma_poll_jfc()` 返回值用 `!= 0` 检查 → 编译正常，逻辑错误
- WRITE_IMM 用错字段（`imm_data` vs `notify_data`）→ 编译正常，即时数据丢失

阶段 2 以单文件视角逐章验证，阶段 4 以整个项目视角检查跨文件问题——这些是编译器可能不报错但运行时会出错的语义缺陷。阶段 4 聚焦：跨文件一致性（头文件→源文件类型同步、共享变量声明一致）、资源归属与完整生命周期链、Verbs 残留扫描、import/unimport 配对检查等。

### 2.5 验证机制的设计逻辑

v2.1 将验证从固定格式模板改为**结果约束**，但验证的严格性不变：

- **必须验证**：patterns.md 的每个章节、pitfalls.md 的每个章节
- **必须达到的状态**：每项判断为符合或 N/A（需说明原因）；FAIL 不可遗留
- **编号动态引用**：P-XX 对应 patterns.md 章节号，PIT-XX 对应 pitfalls.md 章节号，参考文件是唯一权威
- **可审计**：逐项列出验证结果与判断依据，不得以"已验证"替代

v2.0 的固定格式模板（`P-01: PASS/FAIL/N/A - [原因]`）是"怎么做"——规定了 Agent 的输出排版。v2.1 保留"每项必须检查、FAIL 必须修复"这些结果约束，释放格式约束，让 Agent 用最自然的方式输出可审计的验证结果。

### 2.6 映射追溯的设计逻辑

v2.1 新增映射追溯机制，解决阶段间闭环缺失问题。

**问题**：阶段 1 产出映射计划后，阶段 2 逐文件转换时容易遗漏非核心路径的映射项（如 `ibv_create_comp_channel → urma_create_jfce`）。根因是阶段 2 的验证只对照 patterns/pitfalls（通用知识），不对照阶段 1 的映射计划（项目特定清单）——映射计划在获得用户确认后成为"一次性展示物"，失去了作为转换完整性校验基准的价值。

被遗漏的映射项通常有几个特征：
- 非核心路径（属于初始化/清理而非主循环）
- 使用点分散（创建、使用、销毁分布在不同代码区域）
- 条件分支中（如 `if (use_event)` 包裹的事件模式代码）

**设计选择：两层追溯而非一层全局追溯**：

| 层级 | 时机 | 作用 | 为什么不能只用另一层 |
|------|------|------|---------------------|
| 文件级 | 每个文件转换后立即核对 | 核对该文件中直接出现的每个 verbs API 是否已处理 | 项目级追溯太晚，全部转完再查时细节已模糊 |
| 项目级 | 所有文件转换后汇总核对 | 核对阶段 1 映射表每一项是否至少在一个文件中被处理过 | 文件级追溯只看当前文件，跨文件的映射项（如 .h 类型定义被多 .c 引用）可能在某文件中被遗漏 |

两层追溯形成闭环：文件级是过程保障（逐文件闭环），项目级是兜底检查（防遗漏）。这与 patterns/pitfalls 验证的节奏一致——逐文件验证 + 阶段 4 项目级审查。

## 3. 架构

### 3.1 文档层次与职责

skill 采用**一编排 + 四参考**的分层结构：

```
                             SKILL.md
                         编排层 (Orchestrator)
                       ┌────────────────────────┐
                       │ · 触发条件定义          │
                       │ · 迁移硬约束（6 条）    │
                       │ · 4 阶段交付物与约束    │
                       │ · 验证完成条件          │
                       │ · 知识贡献流程          │
                       └───────────┬────────────┘
                                   │ 引用
           ┌──────────────┌──────────────┌──────────────┐
           ▼              ▼              ▼              ▼
    ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐
    │ mapping.md │ │ patterns.md│ │ pitfalls.md│ │urma_sample │
    │            │ │            │ │            │ │   .md      │
    │  查找层    │ │  实现层     │ │  防错层     │ │  规范层    │
    └────────────┘ └────────────┘ └────────────┘ └────────────┘
```

各层职责如下：

**SKILL.md — 编排层**

| 内容 | 说明 |
|------|------|
| 触发条件 | 定义何时激活此 skill |
| 迁移硬约束 | 6 条全过程适用的约束（原始文件不可变、4 阶段不可跳过等） |
| 4 阶段工作流 | 每阶段定义目标、必须交付物、硬约束 |
| 映射追溯 | 两层追溯（文件级 + 项目级），防止映射项遗漏 |
| 阶段检查点 | 阶段 1 后展示映射计划、阶段 4 后展示审查结果，等待用户确认 |
| 验证完成条件 | 动态编号（P-XX / PIT-XX），每项必须检查，FAIL 不可遗留 |
| 知识贡献流程 | 迁移中发现新知识时的反馈机制 |

**mapping.md — 查找层**

| 内容 | 说明 |
|------|------|
| 无 URMA 等价物清单 | 应直接删除的 Verbs API（如 `ibv_alloc_pd()`）及删除注意事项 |
| API 函数映射 | Verbs 函数 → URMA 函数（如 `ibv_open_device` → `urma_create_context`） |
| 枚举映射 | Verbs 枚举 → URMA 枚举（如 `IBV_MTU_1024` → `URMA_MTU_1024`） |
| 结构体字段映射 | Verbs 结构体字段 → URMA 结构体字段（如 `wc.qp_num` → `cr.local_id`） |
| 错误码映射 | URMA 错误码含义，按返回值类别分类（资源管理类/轮询等待类/指针返回类） |
| 常见字段映射错误 | 最易出错的字段混淆（如 `sge.length` → `sge.len`，不是 `length`） |
| 连接建立决策树 | `ibv_modify_qp(RTR/RTS)` → 按 RC/RM/UM 确定正确操作流程 |
| 地址交换格式决策 | `lid:qpn:psn:gid` → `jpn:eid` 的打包内容 |
| 清理顺序决策 | 按 RC/RM/UM 确定正确的资源释放顺序 |
| URMA 特有参数 | Verbs 中无对应项的 URMA 参数（如 `max_rsge`）及其设备能力来源 |

### 3.2 文档间的调用关系

Agent 在迁移过程中的典型查阅路径（建议，非强制）：

```
阶段1 (准备):
  交付物: 源文件分类清单 + API 映射表 + 传输模式判断 + 用户确认
  约束: 系统头文件已读取 | 每 API 已查 mapping | 无法映射已确认删除
  查阅: patterns.md §1 → mapping.md → pitfalls.md

阶段2 (转换):
  交付物: urma_output/ 目录 + 每文件验证结果 + 文件级映射追溯 + 项目级映射追溯汇总 + 构建配置更新
  约束: 原始文件不变 | 头文件类型跨文件一致 | 验证每项已检查无 FAIL 遗留 | 映射表每项已处理
  查阅: mapping.md (API 替换 + 决策树) + patterns.md (代码写法) → pitfalls.md (逐项检查)

阶段3 (编译):
  交付物: 编译成功 + 链接正确
  约束: 无未定义引用、类型不匹配
  查阅: mapping.md §常见字段映射错误 (诊断) → 系统头文件 (权威签名)

阶段4 (审查 - 项目视角):
  交付物: 跨文件一致性确认 + 资源生命周期完整性 + Verbs 残留为零 + 用户确认
  约束: 7 项项目视角检查全部完成 | 修复后重编译
  查阅: 跨文件头文件/源文件同步 | Verbs 残留扫描 | import/unimport 配对
```

### 3.3 信息权威性层级

当文档内容与实际不一致时，按以下优先级决策：

```
系统头文件 (urma_api.h / urma_types.h)    ← 最高权威
       ↓ 若无冲突则参考
urma_sample.md (规范参考实现)              ← 来自官方 umdk 仓库
       ↓ 若无冲突则参考
mapping.md / patterns.md / pitfalls.md     ← skill 自有知识库
       ↓ 若无冲突则参考
Agent 自身的 URMA 知识                     ← 最低权威，可能过时或编造
```

SKILL.md 阶段 1 的硬约束要求先找到并读取系统头文件，就是为了在最早期确立权威来源，避免 Agent 凭"记忆"使用不存在的 API。

## 4. 使用流程

### 4.1 环境前置条件

| 条件 | 说明 |
|------|------|
| URMA 头文件 | `urma_api.h`, `urma_types.h`, `urma_opcode.h`（常见位置 `/usr/include/ub/umdk/urma/`） |
| URMA 库文件 | `liburma.so`, `liburma_common.so`（常见位置 `/lib64/`） |
| C 编译器 | gcc 或兼容编译器 |
| make | 用于构建验证 |

**如果头文件/库未找到**：要求用户安装 URMA 或提供自定义路径。**在没有头文件的情况下不应继续迁移**——Agent 缺乏权威 API 来源，会凭推测编造不存在的函数签名。

### 4.2 触发方式

以下措辞会触发此 skill：

- 提到 "verbs"、"libibverbs"、"ibv_*" 函数
- 提到 "RDMA"、"InfiniBand"、"RoCE"
- 想将 InfiniBand 代码迁移到 URMA
- 提到 "urma" 和 "迁移"/"转换"/"移植"/"翻译"
- 想将 RDMA 应用程序转换为使用 URMA

### 4.3 阶段 1：准备

**目标**：理解源代码并建立完整的 API 映射。

**必须交付物**：
1. 源文件分类清单（需转换 vs 原样复制）
2. Verbs API 映射表（替换 / 删除 / 待确认）
3. 传输模式与连接模型判断
4. 用户确认

**硬约束**：
- 系统头文件已找到并读取
- 每个 verbs API 均已在 mapping.md 中查找过等价物
- 无法映射的 API 已确认删除或标注待确认
- 已了解 URMA 完整资源生命周期

### 4.4 阶段 2：转换

**目标**：在 `urma_output/` 目录中创建转换后的代码，原始文件保持不变。

**必须交付物**：
1. `urma_output/` 目录，保留原始目录结构
2. 每个文件的验证结果
3. 每个文件的映射项执行追溯
4. 项目级映射追溯汇总
5. 构建配置已更新

**转换约束**：
- 原始文件不可修改
- 头文件类型定义变更必须与所有引用它的源文件一致
- 构建文件在所有代码文件转换完成后再更新

**映射追溯**（防止遗漏映射项）：

阶段 1 的映射计划在获得用户确认后不应成为"一次性展示物"，而应作为阶段 2 转换完整性的校验基准。两层追溯确保每个映射项都被执行：

- **文件级追溯**：每个文件转换后，核对该文件中直接出现的每个 verbs API 是否已处理（包括条件分支内、辅助函数中、初始化/清理路径中的调用）
- **项目级追溯**：所有文件转换后，核对阶段 1 映射表每一项是否至少在一个文件中被处理过，不得有"未处理"项

**验证完成条件**：
- patterns.md 和 pitfalls.md 的每个章节已逐项检查
- 每项判断为符合或 N/A（需说明原因）
- FAIL 项已修复，无遗留
- 验证结果可审计（逐项列出判断依据）

### 4.5 阶段 3：编译验证

**目标**：编译、链接并验证转换后的代码。

**必须交付物**：
1. 编译成功（无错误）
2. 链接正确（依赖 `liburma.so` 和 `liburma_common.so`）

**硬约束**：
- 编译通过，无未定义引用、类型不匹配
- 链接验证确认依赖正确的 URMA 库
- 编译错误已排查并修复

### 4.6 阶段 4：审查与优化（强制性）

**目标**：以项目视角检查跨文件语义问题，验证编译之外的正确性。

**必须交付物**：
1. 跨文件一致性确认
2. 资源生命周期完整性
3. Verbs 残留为零
4. 运行时语义正确性
5. 用户确认的审查结果摘要

**7 项项目视角检查**：
1. 头文件→源文件一致性
2. 共享变量一致性
3. 资源归属与生命周期
4. Verbs 残留扫描
5. 重新审视 N/A 项
6. 清理路径完整性
7. 资源泄漏检查

修复并重编译，直到干净。

### 4.7 最终输出结构

```
project/
├── (原始文件 - 不变)
└── urma_output/
    ├── (转换后的 .c 文件)
    ├── (转换后的 .h 文件)
    ├── (原样复制的非 verbs 文件)
    └── Makefile
```

## 6. 已知局限

### 6.1 URMA 文档内容有限，Agent 无法充分理解 URMA 语义

本 skill 的参考文档（mapping/patterns/pitfalls/sample）是 Agent 获取 URMA 知识的主要来源。但 URMA 本身的公开文档体系尚不完善，skill 的参考文档也仅覆盖了常见场景。Agent 缺乏对 URMA 内部机制的深层理解，可能在非标准场景下写出**语法正确但语义不当**的代码——例如选择了不合适的 token_policy、未考虑特定设备的限制、或在并发场景下使用了错误的同步策略。

**缓解措施**：阶段 1 的硬约束要求读取系统头文件作为权威来源；阶段 4 全量审查用于捕获语义问题；用户应审核 Agent 的关键设计决策。

### 6.2 Agent 可能不遵照执行指引

LLM Agent 的固有特性：指令再明确也可能被部分跳过或简略执行。在本 skill 的实践中观察到的典型偏离：

- 跳过阶段 4，直接在阶段 3 编译通过后结束
- 不输出验证结果，或验证结果中写"已验证"而非逐项检查
- 不按参考文件实际章节逐项验证，只检查几个"关键"章节
- 修改原始源文件而非输出到 urma_output/
- 遗漏映射项（阶段 1 映射表中的函数在实际转换时被忽略）

**缓解措施**：v2.1 将约束从步骤中提取出来集中呈现（6 条硬约束），并用"必须交付物"替代步骤序列，使 Agent 更容易理解**终点是什么**而非**路径是什么**。v2.1 新增映射追溯机制（文件级 + 项目级），将阶段 1 的映射计划从"一次性展示物"变为"贯穿阶段 2 的完整性基准"，防止映射项遗漏。但这些是软约束，无法从根本上杜绝跳过行为。**用户应关注 Agent 是否完整交付了每个阶段的必须交付物，特别是映射追溯和阶段 4 的跨文件审查执行情况。**

### 6.3 单进程内迁移，不覆盖跨进程资源协调

skill 的转换逻辑基于单进程内的 URMA 资源生命周期。对于多进程间共享 URMA 资源（如共享 JFR 跨 fork）、跨进程 token 协调、分布式 URMA 应用中的全局状态一致性问题，skill 未提供专门指导。

### 6.4 网络拓扑调优依赖实际部署环境

URMA 支持多种传输模式（RC/RM/UM）和传输协议（RTP/CTP/UTP），选择取决于网络拓扑、设备类型（普通设备 vs bonding 设备）、性能需求等。skill 提供了各模式的代码模式，但不提供"应选哪种模式"的决策建议——这需要对实际部署环境的了解。

### 6.5 URMA API 版本更新后参考文档的滞后风险

参考文档基于 URMA API 25.12.0 编写。当 URMA API 发生变更（函数签名调整、结构体字段变更、新增 API）时，参考文档可能尚未更新。Agent 若同时读取了新版系统头文件，可能发现与 mapping.md 不一致的情况。

**应对策略**：以系统头文件为最高权威（见 §3.3）；发现不一致时应更新参考文档（见 §5）。

## 7. 迭代历史

### v1.0 — 从"怎么做"到"是什么"

**核心变更**：SKILL.md 编排层从步骤序列重构为交付物+约束。

| 变更项 | 变更前 (v0.2) | 变更后 (v1.0) |
|--------|-------------|-------------|
| 阶段定义 | 编号步骤序列（1.1→1.2→1.3→1.4） | 目标 + 必须交付物 + 硬约束 |
| 文件操作 | 具体 bash 命令（`find`、`mkdir`、`cp`、`make`） | 信息需求描述 + 目录结构约束 |
| 验证清单 | 固定格式模板（`P-XX: PASS/FAIL/N/A`） | 验证完成条件（每项已检查、无未修复 FAIL） |
| 转换顺序 | 硬性序列（头文件→主源文件→工具文件→构建文件） | 灵活约束（头文件类型一致性、构建文件最后更新） |
| 检查点 | 完整 markdown 模板 | 必须包含的信息项 |
| 硬约束 | 散落在步骤描述中 | 集中呈现于工作流之前（6 条） |
| 映射追溯 | 无（映射计划是一次性展示物） | 两层追溯（文件级 + 项目级），映射计划成为完整性基准 |
| 构建配置 | 含完整 Makefile 模板 | 简化为必须更新项 |

**设计动机**：基于 skill 编写经验——"给 Agent 定义'是什么'，而非'怎么做'，保证过程灵活，结果稳定"。v0.2 的步骤序列是过程约束，限制了 Agent 的灵活性；v1.0改为结果约束，只要交付物满足条件，Agent 可自行决定执行路径。参考文件（mapping/patterns/pitfalls）本身是声明式知识，无需改动。

**映射追溯的动机**：实际迁移中发现 Agent 会在阶段 1 列出映射项（如 `ibv_create_comp_channel → urma_create_jfce`），但阶段 2 实际转换时遗漏该函数。根因是阶段 1 映射计划与阶段 2 转换之间缺少闭环——验证只对照通用知识（patterns/pitfalls），不对照项目特定清单。两层追溯将映射计划从"一次性展示物"变为"贯穿阶段 2 的完整性基准"。

### v0.2 — 精简与验证机制增强

**核心变更**：消除冗余与重复，强化验证机制。

- 将 SKILL.md 和 references 三份文档改为中文
