# C 规范速查

本文件是面向代码编写、检视、审阅的 C 语言编程规范精简工作摘录。

内容为规范要点转述，保留规则编号，方便审阅评论引用。

## 总体原则

- 清晰第一：代码首先是给人读的；除非性能已被证明是瓶颈，通常可读性优先于性能。
- 简洁为美：删除废弃代码，抽取重复代码，优先编写小而直接的函数。
- 修改旧代码时保持原有风格；除非用户要求严格规范化，或原有风格违反更强项目规则。
- “规则”为强制约定；“建议”应认真考虑，除非存在明确的本地理由。

## 头文件

- 原则 1.1：头文件放接口声明，不放实现。私有函数、私有宏、私有枚举、私有结构和变量定义不要放入公共头文件。
- 原则 1.2：头文件职责单一。
- 原则 1.3：依赖应指向更稳定的接口或模块。
- 规则 1.1：每个 `.c` 文件应有同名 `.h` 文件用于公开接口；入口文件或没有公开接口的文件可例外。
- 规则 1.2：禁止头文件循环依赖。
- 规则 1.3：`.c` 和 `.h` 文件禁止包含用不到的头文件。
- 规则 1.4：头文件应自包含：使用者只包含该头文件即可编译，不应依赖额外包含顺序。
- 规则 1.5：总是使用 include guard。优先使用唯一名称，如 `PROJECT_PATH_FILE_H`；不要以下划线开头或结尾。
- 规则 1.6：禁止在头文件中定义变量。
- 规则 1.7：使用其他 `.c` 文件提供的接口时必须包含其头文件，禁止在 `.c` 中直接写外部函数或变量的 `extern` 声明。
- 规则 1.8：禁止在 `extern "C"` 块中包含头文件。
- 建议 1.1：包含多个 `.c` 文件的模块可提供一个按模块命名的对外头文件。
- 建议 1.2：模块包含多个子模块时，每个子模块可提供自己的对外头文件。
- 建议 1.3：避免使用 `.inc` 等非习惯头文件扩展名。
- 建议 1.4：同一产品内头文件包含顺序保持一致。

## 函数

- 原则 2.1：一个函数只完成一件功能明确的事情。
- 原则 2.2：重复代码应抽取成函数；重复两次就考虑抽取，超过三次应立即着手消除。
- 规则 2.1：新增函数不超过 50 行非空非注释行。
- 规则 2.2：新增函数代码块嵌套不超过 4 层。
- 规则 2.3：可重入函数应避免共享变量；不可避免时必须加互斥保护。
- 规则 2.4：参数合法性检查由调用者还是接口函数负责，应在模块或项目内统一。
- 规则 2.5：对函数错误返回码要全面处理。
- 规则 2.6：设计高扇入、合理扇出；扇出尽量小于 7。
- 规则 2.7：及时清除未调用的函数和变量。
- 建议 2.1：不变参数使用 `const`。
- 建议 2.2：尽量避免函数使用全局变量、静态局部变量和 I/O；不可避免时集中使用。
- 建议 2.3：检查非参数输入的有效性，如数据文件、公共变量等。
- 建议 2.4：函数参数个数不超过 5 个；超过时考虑拆分函数或封装数据。
- 建议 2.5：除打印类函数外，避免可变长参数函数。
- 建议 2.6：源文件范围内的函数，除非需要外部可见，否则加 `static`。

## 命名

- 规则 3.1：产品或项目组内部保持统一命名风格。
- 规则 3.2：全局变量应加 `g_` 前缀。
- 规则 3.3：静态变量应加 `s_` 前缀。
- 规则 3.4：禁止单字节变量名，但允许 `i`、`j`、`k` 作为局部循环变量。
- 规则 3.5：数值或字符串常量建议使用全大写字母，单词间用下划线。
- 规则 3.6：除头文件保护符、编译开关等特殊标识外，宏定义不能以下划线开头或结尾。
- 建议 3.1：有互斥含义的变量或相反动作的函数使用正确反义词组。
- 建议 3.2：避免名字中出现数字编号，除非数字有逻辑含义。
- 建议 3.3：标识符前不应泛化添加产品、模块、部门名称前缀。
- 建议 3.4：平台或驱动适配代码可保持平台/驱动自身命名风格。
- 建议 3.5：重构或修改已有代码时，保持原有命名风格。
- 建议 3.6：文件名统一使用小写字符。
- 建议 3.7：不建议使用匈牙利命名法。
- 建议 3.8：变量使用名词或形容词加名词命名。
- 建议 3.9：函数按动作命名，通常使用动词或动词加名词。
- 建议 3.10：函数指针除前缀外按函数命名规则命名。

## 变量与数据

- 规则 4.1：防止局部变量与全局变量同名。
- 规则 4.2：通信结构必须注意字节序。
- 规则 4.3：严禁使用未初始化变量作为右值。
- 建议 4.1：降低全局变量耦合度，让一个模块或函数负责修改/创建，其余模块只访问。
- 建议 4.2：通过 API 访问模块数据，避免直接暴露内部数据。
- 建议 4.3：变量首次使用前初始化，初始化位置尽量靠近使用位置。
- 建议 4.4：明确全局变量初始化顺序，避免跨模块初始化依赖。
- 建议 4.5：尽量减少不必要的隐式转换和强制转换。

## 宏与常量

- 规则 5.1：宏定义表达式时，参数和整体表达式都要使用完备括号。
- 规则 5.2：多语句宏应安全包裹，通常使用 `do { ... } while (0)`。
- 规则 5.3：使用宏时不允许宏参数发生变化。
- 规则 5.4：不允许直接使用魔鬼数字。
- 建议 5.1：除非必要，尽量使用函数代替宏。
- 建议 5.2：常量尽量使用 `const` 定义代替宏。
- 建议 5.3：宏定义中尽量不使用 `return`、`goto`、`continue`、`break` 等改变流程的语句。

## 质量与正确性

- 规则 6.1：禁止内存操作越界。
- 规则 6.2：禁止内存或其他资源泄漏。
- 规则 6.3：禁止引用已经释放的内存空间。
- 规则 6.4：防止差 1 错误。
- 规则 6.5：所有 `if ... else if` 结构应由 `else` 结束；`switch` 必须有 `default` 分支。
- 建议 6.1：函数中分配的内存应在函数退出前释放，除非所有权已明确转移。
- 建议 6.2：`if` 语句尽量加 `else` 分支；没有 `else` 时应谨慎确认。
- 建议 6.3：不要滥用 `goto`；仅在能改善结构化清理或错误处理时使用。
- 建议 6.4：时刻注意表达式是否会上溢或下溢。

## 程序效率

- 建议 7.1：将循环不变计算移到循环外。
- 建议 7.2：对于多维大数组，按有利于局部性的顺序访问。
- 建议 7.3：在频繁分配和释放成本明显时使用资源池。
- 建议 7.4：小而高频调用的函数可考虑 `inline` 或宏，但要权衡可调试性、代码大小和类型检查。

## 注释

- 原则 8.1：优秀代码可以自我解释；注释不能把糟糕代码变好。
- 原则 8.2：注释内容要清楚、准确、无二义性。
- 原则 8.3：注释解释意图和约束，不重复描述显而易见的代码。
- 规则 8.1：修改代码时维护周边注释，删除过期注释和注释掉的废弃代码。
- 规则 8.2：文件头部注释应包含版权、版本、日期、作者/工号、内容、功能、与其他文件关系、修改日志；头文件注释还应简要说明函数功能。
- 规则 8.3：函数声明处注释说明功能、性能、用法、输入输出参数、返回值、可重入要求；定义处在需要时说明重要实现要点。
- 规则 8.4：全局变量要有较详细注释，说明功能、取值范围和访问注意事项。
- 规则 8.5：注释放在相关代码上方相邻位置或右方，不放在下面。
- 规则 8.6：`switch` 中刻意 fallthrough 必须在进入下一个 `case` 前加明确注释。
- 规则 8.7：注释中避免使用缩写，除非是业界通用或子系统内标准化缩写。
- 规则 8.8：同一产品或项目组统一注释风格。
- 建议 8.1：避免在一行代码或表达式中间插入注释。
- 建议 8.2：注释语言保持一致；中英混合团队建议多用中文，除非产品另有规定。
- 建议 8.3：文件头、函数头、全局常量变量、类型定义的注释建议使用工具可识别格式，如 Doxygen。

## 排版与格式

- 规则 9.1：使用缩进风格，每级缩进 4 个空格；Tab 转空格。
- 规则 9.2：相对独立的程序块之间、变量说明之后必须加空行。
- 规则 9.3：语句不能过长；按项目定义行宽换行。换行时增加一级缩进，优先在低优先级操作符处换行。
- 规则 9.4：一行只写一条语句。
- 规则 9.5：`if`、`for`、`do`、`while`、`case`、`switch`、`default` 等语句独占一行。
- 规则 9.6：双目操作符前后加空格，逗号和分号后加空格；`->`、`.`、单目 `!`、`~`、
  `++`、`--`、取地址 `&`、解引用 `*` 前后不加空格；`if (`、`for (`、`while (`、
  `switch (` 关键字后加空格。
- 建议 9.1：注释符和注释内容之间用一个空格分隔。
- 建议 9.2：关系紧密的代码尽量相邻。

## 表达式

- 规则 10.1：表达式的值不能依赖 C 标准未指定的求值顺序。把有副作用的运算拆成独立语句。
- 建议 10.1：函数调用不要作为另一个函数的参数使用，尤其当影响阅读、调试或隐藏求值顺序问题时。
- 建议 10.2：赋值语句不要写在 `if` 条件或函数参数中。
- 建议 10.3：混合操作符或转换时用括号明确顺序；简单表达式不要过度加括号。
- 建议 10.4：赋值操作符不能用在产生布尔值的表达式上。

## 构建与静态检查

- 规则 11.1：使用编译器最高告警级别；理解并通过修改代码消除告警，不通过降低告警级别消除。
- 规则 11.2：产品/项目中统一编译开关、静态检查选项和告警清除策略；必须局部屏蔽告警时写清楚原因。
- 规则 11.3：本地构建或静态检查配置应和持续集成一致。
- 规则 11.4：只提交通过本地构建且不会破坏集成的代码。
- 建议 11.1：小心使用编辑器块拷贝功能编程。

## 可测性与单元测试

- 原则 12.1：模块划分清晰、接口明确、耦合小、有明确输入输出，才便于单元测试。
- 规则 12.1：产品/项目应有统一的集成测试和系统联调用调测开关及打印函数。
- 规则 12.2：调测日志应统一分类、级别、配置方式，并包含关键上下文，如文件/模块、行号、函数、被调函数、错误码和环境。
- 规则 12.3：使用断言记录内部假设。
- 规则 12.4：不能用断言检查运行时错误或外部输入；这些场景必须用正常防错代码处理。
- 建议 12.1：为单元测试和故障注入测试准备方法和通道。
- 规则 14.1：在编写代码的同时或之前编写单元测试，验证设计和编码正确性。
- 建议 14.1：单元测试尽量关注单元行为，而不是实现细节。

## 安全性

- 原则 13.1：检查用户或外部输入，尤其当其作为循环条件、数组下标、内存分配尺寸、格式串、命令参数、SQL/业务数据或持久化数据时。
- 规则 13.1：确保所有字符串以 NUL 结束。
- 规则 13.2：不要将边界不明确的字符串写入固定长度数组。
- 规则 13.3：避免整数上溢和下溢。
- 规则 13.4：避免符号错误。
- 规则 13.5：避免大整数类型转换为小整数类型时发生截断错误。
- 规则 13.6：确保格式字符和参数数量、类型匹配。
- 规则 13.7：避免将用户输入作为格式化字符串的一部分或全部。
- 规则 13.8：避免使用 `strlen()` 计算二进制数据长度。
- 规则 13.9：需要和 `EOF` 比较时，使用 `int` 类型变量接收字符 I/O 函数返回值。
- 规则 13.10：防止命令注入；避免对拼接用户输入使用 `system()`/`popen()`，可用时优先使用 exec 类 API。

## 可移植性

- 规则 15.1：不能定义、重定义或取消定义标准库/平台保留的标识符、宏和函数。
- 建议 15.1：标准语句能满足需求时，避免使用和硬件或操作系统关系很大的语句。
- 建议 15.2：除特殊需求外，避免使用嵌入式汇编。
